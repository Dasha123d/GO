# Шаблоны real-time взаимодействия

Рассмотрим типичные архитектурные решения при использовании WebSocket.

## Чат-комнаты

Модель "комнат": сервер хранит карту комнат, каждая комната — множество клиентских соединений. При получении сообщения от клиента оно транслируется всем остальным в комнате.

```go
type Room struct {
    clients map[*websocket.Conn]bool
    broadcast chan []byte
    mu sync.Mutex
}
```
Для каждого соединения запускаются горутины чтения и записи. Писатель читает из канала `broadcast` и отправляет всем клиентам. Это классический паттерн, см. пример `chat-server.go`.

## Широковещательная рассылка (fan-out)
Если нужно отправить одно сообщение всем подключённым клиентам (не разделяя на комнаты), используют единый центральный хаб, который хранит все активные соединения. При получении события хаб итерирует и пишет в каждое соединение.

## Pub/Sub с внешним брокером
Интеграция с Redis Pub/Sub или NATS позволяет масштабировать WebSocket-сервер горизонтально: сервер подписывается на канал и пересылает сообщения клиентам. Клиент может подписаться на конкретный канал, указав его в сообщении.

Пример: клиент отправляет `{"action":"subscribe","channel":"news"}`, сервер регистрирует соединение в слушатели этого канала. При поступлении сообщения из Redis/NATS сервер доставляет его нужным клиентам.

## Управление состоянием клиентов
Важно корректно добавлять и удалять соединения из коллекций, иначе будут утечки памяти и попытки записи в закрытые сокеты. Используйте каналы для регистрации/дерегистрации или мьютексы.

Типичная структура хаба:
```go
type Hub struct {
    clients    map[*Client]bool
    broadcast  chan []byte
    register   chan *Client
    unregister chan *Client
}
```
Главный цикл `run()` слушает каналы и мутирует карту.
